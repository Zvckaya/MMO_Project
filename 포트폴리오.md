# MMO Game Server Portfolio

Windows IOCP 기반 고성능 비동기 TCP 게임 서버.  
네트워크 레이어(`IOCPServer`)를 직접 구현하고 그 위에 게임 서버(`GameServer`)를 올리는 레이어 분리 구조로 설계.

**빌드 환경**: Visual Studio 2022, C++20, Windows 11 x64  
**스트레스 테스트**: 100 클라이언트 / OverSend 200 / Disconnect Delay 0 / 300초 — Packet Error 0

## 1. 아키텍처

```
[ GameServer ] ─── IOCPServer 상속, virtual 콜백만 구현
      │
[ IOCPServer ] ─── 네트워크 레이어 (소켓, IOCP, 세션 생명주기)
      │
[ Session 배열 ] ─── std::array<Session, MAXSESSIONSIZE>, 서버 시작 시 전체 생성
      │
[ RingBuffer ] ─── Session당 recvBuffer / sendBuffer 각 1개
```

**레이어 분리 원칙**
- 네트워크 레이어와 콘텐츠 레이어는 포인터를 직접 교환하지 않음
- 인터페이스: `SessionID + Packet*` 만 경계를 넘고, 응답은 `SendPacket(SessionID, Packet*)` / `Disconnect(SessionID)` 만 사용

**SessionID 구조**
```
[ 상위 48bit: generation ] [ 하위 16bit: slot index ]
```
슬롯 재사용 시 generation이 달라지므로 이전 세션의 ID로 전송을 시도하면 자동으로 드롭됨.

## 2. 구현 기능

### 2.1 스레드 구조

총 4종류의 스레드가 역할을 나눠 게임 루프를 구성한다.

| 스레드 | 개수 | 역할 |
|---|---|---|
| Worker (GQCS) | N개 | IOCP 완료 통지 처리, 패킷 파싱 |
| ImmediateThread | 4개 | per-session JobQueue 소비, 패킷 디스패치 |
| FrameThread | 1개 | 맵/플레이어 상태 갱신, 브로드캐스트 |
| DBThread | 1개 | DB 요청 큐 소비 |

**ImmediateThread — Actor 모델**

세션마다 독립적인 `JobQueue`를 보유. 패킷이 들어오면 해당 세션의 큐에 job을 등록하고, ImmediateThread가 `GlobalQueue`에서 `JobQueue`를 꺼내 `Execute(endTick)` 실행. 같은 세션의 패킷은 항상 단일 스레드에서 순차 처리되어 세션 내 순서가 보장됨.

**FrameThread**

`_maps`, `_players`, `_sessionToMapID`를 단독 소유. 이 데이터에 접근하는 스레드가 하나뿐이라 별도 락이 없음. 이동, 전투, 아이템 등 타 플레이어 상태가 필요한 모든 게임 로직은 `FrameTask` 큐를 통해 FrameThread에 위임.

**판단 기준**
```
내 세션 상태만 변경  → ImmediateThread (JobQueue)
맵·타인 상태 변경    → FrameThread (FrameTask)
```

### 2.2 인증 시스템

```
클라이언트 → CS_LOGIN_AUTH(ticket 32byte)
  → AuthThread → HTTP POST /verify → AuthResult
  → DBThread → loadPlayer → PlayerDBData
  → FrameThread → playerAuth FrameTask → Player 생성
```

- 인증 전 `PKT_CS_LOGIN_AUTH` 외 패킷은 전부 차단 후 즉시 Disconnect
- 인증 상태는 per-session `JobQueue::isAuthenticated` (atomic bool)에 보관 — 패킷 수신마다 글로벌 뮤텍스 없이 O(1) 체크

### 2.3 이동 시스템

**Click-to-move float 좌표**

```
클라 → CS_MOVE(curX, curY, destX, destY)
  → ImmediateThread: pendingMove에 저장, dirty = true
  → FrameThread Update(): dirty 확인 → Player 목적지 갱신 → SC_MOVE BroadcastExcept
  → 이후 매 프레임: speed × (deltaMs/1000.f) 로 서버 내부 위치만 갱신
```

- **속도 검증**: `curX/curY` 와 서버 `posX/posY` 거리를 `speed × elapsed × 1.2f` (20% 허용 오차)와 비교 → 초과 시 `SC_MOVE_CORRECT` 강제 보정
- **정지 처리**: `CS_STOP(curX, curY)` → `pendingStop` → FrameThread에서 서버 좌표 덮어쓰기, 클라/서버 위치 DEBUG 로그 출력

### 2.4 전투 시스템

```
CS_ATTACK(targetID) → FrameTask(playerAttack)
  → 공격자·피격자 포인터 조회
  → 거리 검증 (ATTACK_RANGE = 2.0f)
  → 데미지 처리 (hp -= damage)
  → SC_ATTACK BroadcastAll(attackerID, targetID, damage, targetHp)
```

서버에서 공격 범위를 재검증하므로 클라이언트 측 위치 조작을 차단.

### 2.5 맵 시스템

`GameMap` 객체가 맵별 플레이어 목록과 월드 아이템 목록을 소유. FrameThread가 `_maps`를 단독 관리하므로 락 불필요.

**접속 시 패킷 순서**
```
SC_MAP_INFO → SC_ITEM_APPEAR × N → SC_INVENTORY_UPD × N
→ SC_CREATE_MY_CHARACTER → SC_CREATE_OTHER_CHARACTER × N → SC_WORLD_ENTER
```

**맵 이동**
```
CS_MAP_CHANGE_REQ(targetMapID) → FrameTask(playerMapChange)
  → oldMap에서 TakePlayer() (unique_ptr 소유권 이전)
  → BroadcastExcept(oldMap, SC_DESPAWN)
  → newMap->AddPlayer()
  → SC_MAP_CHANGE(result=0) + 재진입 패킷 시퀀스
```

`TakePlayer()` 로 소유권을 명시적으로 이전해 DESPAWN 브로드캐스트 중 댕글링 포인터를 방지.

### 2.6 아이템 시스템

**서버 시작 시 아이템 시트 로드**
```cpp
DBClient db;
db.Connect(...);
_itemDataMap = db.LoadItemData(); // unordered_map<uint16_t, ItemData>
```
스레드 시작 전 동기 로드 → 이후 읽기 전용으로 사용, 동기화 불필요.

**아이템 드롭** (`CS_ITEM_DROP`)
1. 인벤토리 슬롯 클리어 + `SC_INVENTORY_UPD(itemID=0)`
2. `map->SpawnWorldItem()` + `SC_ITEM_APPEAR` BroadcastAll

**아이템 줍기** (`CS_ITEM_PICKUP`)
1. 거리 검증 (ITEM_PICKUP_RANGE = 3.0f)
2. 빈 슬롯 탐색
3. `map->RemoveWorldItem()` + `SC_ITEM_DISAPPEAR` BroadcastAll
4. `SC_INVENTORY_UPD` 본인에게 전송

**슬롯 교환** (`CS_ITEM_MOVE`)
- `std::swap(inventory[from], inventory[to])` 후 두 슬롯 `SC_INVENTORY_UPD` 전송

**월드 아이템 타임아웃**
- `Update()`에서 매 프레임 `GetTickCount64() - spawnTick >= 30000ms` 체크 → 자동 소멸 + `SC_ITEM_DISAPPEAR` BroadcastAll

### 2.7 DB 연동

**비동기 DBThread — fire-and-forget + 콜백 패턴**
```
FrameThread / JobQueue → EnqueueDBRequest() → _dbPendingQueue push
DBThread → 큐 소비 → DB 처리 → (결과 필요 시) onLoad 콜백 → EnqueueFrameTask
```
IO 스레드와 FrameThread는 DB 응답을 기다리지 않음.

**처리 항목**
- `loadPlayer`: 위치, HP, 맵 ID, 인벤토리 전체 로드 (접속 시)
- `savePlayer`: 위치, HP, 맵 ID 저장 (퇴장 시)
- `saveInventory`: DELETE + INSERT 트랜잭션 (퇴장 시, 원자성 보장)

**Ping 기반 재연결**: 요청 처리 전 `db.Ping()` — 연결 끊기면 자동 재접속.

## 3. 트러블슈팅

### 3.1 동기화 / Race Condition

---

#### `_isSending` race condition — Echo 데이터 변조
**날짜**: 2026-04-14

**현상**  
단일 세션이 연속 전송할 때 클라이언트가 수신한 echo 값이 송신값과 달라지는 현상.

**원인**  
`_isSending`이 일반 `bool`이라 atomic하지 않았음. Worker 스레드 A가 `_isSending` 읽기(false 확인)와 쓰기(true 세팅) 사이에 Worker 스레드 B가 끼어들어 같은 소켓에 WSASend가 2개 등록됨. 두 send가 sendBuffer의 동일한 front 위치에서 데이터를 읽어 중복 전송 또는 뒤섞임 발생.

**해결**  
`bool` → `std::atomic<bool>` 교체. `SendPost()` 내부에서 `compare_exchange_strong(expected=false, true)` CAS로 단일 진입 보장.

**배운 점**  
세션 상태 플래그는 반드시 atomic으로 보호해야 한다. CAS를 함수 내부에 두면 호출부가 플래그를 직접 건드리지 않아도 되어 인터페이스가 단순해진다.

---

#### sendBuffer 동시 접근 — `WSASend 10014` + 비정상 len 발생
**날짜**: 2026-04-16

**현상**  
`SendPost()`에서 `WSASend 10014(WSAEFAULT)` + `len=4294947xxx` 같은 비정상 WSABUF 길이가 로그에 찍히며 서버가 먼저 Disconnect를 수행.

**원인**  
`SendPost()`가 `GetUseSize()`, `DirectDequeueSize()`, `GetFrontBufferPtr()`를 각각 따로 읽어 WSABUF를 구성하는 구조였음. ImmediateThread의 `Enqueue`와 Worker의 `MoveFront`가 동시에 실행되면 각 load가 서로 다른 시점의 `front/rear` 값을 읽어 snapshot race가 발생. `WSABUF.len`이 음수 해당 값으로 깨져 `ULONG`으로 해석되면서 비정상 길이가 출력됨.

**해결**  
`sendBuffer` 전체를 `_sendLock`(mutex)으로 보호. `TryEnqueueSend()`, `SendPost()`, `CompleteSend()`를 같은 락 아래에서 처리하도록 정리.

**배운 점**  
`front/rear`를 `std::atomic<int>`로 각각 관리한다고 해서 링버퍼 전체 상태가 thread-safe해지는 것이 아니다. 여러 번 따로 읽는 구조에서는 일관된 snapshot이 보장되지 않는다.

---

#### `Disconnect`와 `Send enqueue` 경쟁 — 잔여 Packet Error
**날짜**: 2026-04-17

**현상**  
극단 부하 300초 테스트에서 `Packet Error`가 산발적으로 발생 (에러율 5.3 × 10⁻⁸). Disconnect/Timeout은 0.

**원인**  
`IOCPServer::SendPacket()`이 sessionID 검증 후 `TryEnqueueSend()`로 진입하기 전 사이 구간에 공통 락이 없었음. 이 틈에 다른 스레드가 `Disconnect()`를 먼저 실행해도 기존 `TryEnqueueSend()`는 enqueue를 허용했음. 이미 끊기기 시작한 세션의 echo가 send 큐에 들어가 클라이언트가 "내가 기대한 값과 다른 응답"을 수신.

**해결**  
`TryEnqueueSend(ownerSessionID)`가 `_sendLock` 안에서 `GetSessionID() == ownerSessionID`, `_isDisconnected == false`, `_sock != INVALID_SOCKET`을 재검증하도록 변경. `Disconnect()`도 같은 `_sendLock` 안에서 `_sock = INVALID_SOCKET`을 먼저 반영해 stale enqueue 창을 원천 차단.

**배운 점**  
상태 검사는 실제 상태를 소비하는 락 안에서 다시 확인해야 의미가 있다. `sessionID` 검사는 슬롯 재사용을 막지만, 같은 세션이 지금 막 disconnect 중인가까지는 보장하지 않는다.

---

#### `SendPost()` 내 Disconnect 호출 데드락
**날짜**: 2026-04-17

**현상**  
고부하 + 버퍼 포화 상황에서 프로세스가 exit code 3으로 즉시 종료.

**원인**  
`SendPost()`가 `_sendLock`을 잡은 채로 WSASend 즉시 에러 시 `Disconnect()`를 호출. `Disconnect()` → `ReleaseRef()` → `ioCount == 1`이면 `Clear()` → `Clear()`가 `_sendLock`을 다시 획득 시도 → 같은 스레드에서 non-recursive mutex 이중 잠금 → UB/데드락.

**해결**  
`needDisconnect` 플래그 패턴 적용. `_sendLock` 블록 안에서 플래그만 세우고, 락 해제 후 `if (needDisconnect) Disconnect()` 호출.

```cpp
bool needDisconnect = false;
{
    std::lock_guard lock(_sendLock);
    // ... WSASend 즉시 에러 시 needDisconnect = true
}
if (needDisconnect) Disconnect();
```

**배운 점**  
락을 잡은 채로 절대 `Disconnect()`/`ReleaseRef()`를 호출하지 않는다. `Clear()`가 그 락을 재획득하면 같은 스레드에서 non-recursive mutex 이중 잠금 → UB.

### 3.2 세션 생명주기

---

#### `ioCount` 기반 세션 생명주기 설계
**날짜**: 2026-04-14

**현상**  
클라이언트 접속 종료 시 세션이 정리되지 않아 시간이 지날수록 세션 슬롯이 고갈됨.

**원인**  
비동기 I/O 특성상 소켓을 닫는 시점에 아직 GQCS 큐에 완료 이벤트가 남아 있을 수 있음. 소켓만 닫고 즉시 슬롯을 반환하면 뒤늦게 도착하는 완료 이벤트가 이미 다른 세션이 사용 중인 슬롯을 건드리는 문제 발생. 추가로 `fetch_sub(1)`이 감소 **전** 값을 반환한다는 점을 몰라 `== 0` 체크로 작성해 Clear()를 놓치는 실수도 있었음.

**해결**  
```
Accept 시              → _ioCount++ (소유권 ref)
RecvPost/SendPost 전   → _ioCount++ (I/O ref)
OnRecv 호출 전         → AddContentRef() = _ioCount++ (콘텐츠 job ref)
완료 루틴 정상 처리    → ReleaseRef()
job 실행 완료          → ReleaseContentRef(sessionID)
첫 에러 감지           → Disconnect() → CAS로 closesocket + 소유권 ref + I/O ref 반납
_ioCount == 0          → Clear()
```

**배운 점**  
소유권 ref 개념이 없으면 정상 동작 중에도 Clear()가 호출된다. `fetch_sub`는 감소 **전** 값을 반환하므로 "0이 됐는지" 확인하려면 `== 1` 체크가 맞다.

#### ImmediateThread job 실행 중 세션 Clear — per-job content ref
**날짜**: 2026-04-16

**현상**  
재접속 폭주 조건에서 `PACKET ERROR RecvPacket: 0x00` 반복 발생.

**원인**  
ImmediateThread가 `SendPacket(sessionID, packet)` → `TryEnqueueSend` → `sendBuffer.Enqueue`를 실행하는 동안 Worker 스레드가 recv 에러 → `Disconnect()` → `Clear()` → `sendBuffer.ClearBuffer()`를 완료하고 슬롯을 반환할 수 있었음. ImmediateThread는 ioCount를 들고 있지 않으므로 Clear()를 막을 방법이 없었음. 새 클라이언트가 같은 슬롯을 재사용하면 ImmediateThread가 쓴 `m_iRear` 값이 잔류해 sendBuffer가 오염된 상태로 시작됨.

**해결**  
Worker 스레드가 `OnRecv()` 호출 전에 `session->AddContentRef()`로 세션 ref를 하나 더 확보. job 람다 끝에서 `ReleaseContentRef(sessionID)` 호출. `AddContentRef()`가 살아있는 동안 `ioCount > 0`이 보장되어 Clear() 진입이 억제됨.

**배운 점**  
recv 응답이든 서버 주도 send든, `SendPacket`을 호출하기 전에 반드시 ioCount ref를 들고 있어야 한다.

---

#### `IOCPServer::Disconnect(SessionID)` ioCount 불균형
**날짜**: 2026-04-19

**현상**  
콘텐츠 레이어(FrameThread)에서 `Disconnect(sessionID)`를 호출하면 뒤늦게 도착한 이전 소켓의 에러 완료 통지가 새 클라이언트를 강제 종료시키는 현상.

**원인**  
`Session::Disconnect()`는 "이 I/O의 ref"를 반납하는 `ReleaseRef()`를 무조건 호출하는 구조 (GQCS 워커가 I/O ref를 들고 호출한다는 전제). `IOCPServer::Disconnect(SessionID)`는 ioCount를 증가시키지 않고 바로 `session->Disconnect()`를 호출하므로, 콘텐츠 레이어에서 호출 시 ioCount가 예상보다 1 더 감소. `closesocket` 직후 ioCount가 0이 되어 Clear() → 슬롯 반환 → 새 클라이언트 슬롯 재사용 → 이전 소켓의 에러 완료 통지가 새 세션에 도달.

**해결**  
`IOCPServer::Disconnect(SessionID)`에서 `session->AddContentRef()`를 먼저 호출해 I/O ref를 미리 확보.

**배운 점**  
ref count 기반 Disconnect는 항상 "호출자가 ref를 하나 들고 있다"는 전제로 설계됨. 외부에서 호출하는 경우 반드시 사전에 ref를 확보해야 한다.

#### 공유 큐 → GlobalQueue + per-player JobQueue 전환 — TPS 3배 상승
**날짜**: 2026-04-16

**현상**  
ImmediateThread N개가 단일 공유 큐를 mutex로 소비하는 구조에서 스레드를 늘려도 TPS가 상승하지 않았음.

**원인**  
모든 스레드가 하나의 mutex에 집중 → 스레드가 많을수록 lock contention이 비례해 증가. 같은 세션의 패킷이 서로 다른 스레드에서 동시에 처리될 수 있어 순서 보장도 없음.

**해결**  
per-session `JobQueue` + `GlobalQueue` 구조 전환.  
- job이 0→1이 되는 순간에만 JobQueue를 GlobalQueue에 등록 → 패킷 1000개가 쏟아져도 GlobalQueue lock은 최대 1회
- 스레드가 JobQueue를 획득한 뒤에는 lock 없이 Execute() → 스레드 추가 시 TPS 선형 증가
- 같은 JobQueue는 동시에 하나의 스레드만 Execute() → 세션 내 패킷 순서 자동 보장
- 64ms 선점(endTick) → 특정 세션 폭주 시 다른 세션이 굶지 않음

**결과**: 동일 클라이언트 수, 동일 스레드 수 조건에서 **TPS 3배 이상 상승** 확인.

#### 락프리 메모리 풀 도입 — TLS 배치(Batch) 캐시 구현
**날짜**: 2026-04-18

**현상**  
패킷 처리량(TPS)이 70만을 넘는 상황에서 매 패킷마다 `new`/`delete` 힙 할당/해제가 병목이 될 수 있다고 판단, 락프리 메모리 풀을 설계.

**설계 핵심**  
- 침습형 노드: `alignas(DATA) unsigned char data[sizeof(DATA)]`를 Node 내부에 embed → `Free(DATA*)` 시 `offsetof(Node, data)`로 포인터 역산, 별도 맵 불필요
- ABA 방지: x64 가상 주소는 48비트만 사용하므로 `atomic<uint64_t> _head`의 상위 16비트를 태그로 사용, Push/Pop마다 단조 증가
- TLS 배치 캐시 (`TLS_CACHE_BATCH = 32`): 글로벌 CAS는 32개 단위로만 발생 → 패킷 1개당 CAS 1회 → 32개당 CAS 1회로 절감
  - 캐시가 64개 이상 쌓이면 32개를 공유 풀로 반납 → TLS 캐시는 항상 32~63개 사이에서 안정적으로 유지

**성능 측정 결과 (Release / 100 클라이언트 / 180초)**
| 방식 | 평균 TPS |
|---|---|
| new/delete | 737,838 |
| MemoryPool (초기 구현) | 443,066 (-39.9%) |

초기 구현이 오히려 느렸던 원인: Worker(Alloc)와 ImmediateThread(Free)가 다른 스레드라 TLS 효과 제한적, CAS spin 오버헤드가 LFH(MSVC Release 기본 힙)를 역전. 배치 TLS 구조 적용 후 CAS 빈도를 줄여 성능 회복.

**배운 점**  
락프리가 무조건 빠른 게 아니다. CAS spin 자체가 캐시 라인 invalidation을 유발하므로 경합이 많은 환경에서는 LFH보다 오히려 느릴 수 있다. 글로벌 접근 빈도 자체를 줄이는 것이 핵심.

---

#### `SO_SNDBUF=0` (Zero-Copy) 제거 — TPS 3배 이상 상승
**날짜**: 2026-04-18

**현상**  
`SO_SNDBUF=0` / `SO_RCVBUF=0` 적용 상태에서 물리 네트워크 스트레스 테스트 결과 TPS가 루프백 대비 1/3 수준으로 저하.

**원인**  
`SO_SNDBUF=0`은 zero-copy를 위해 커널 송신 버퍼를 제거하는 옵션. WSASend마다 IRP(I/O Request Packet) 생성 + 앱 버퍼 page lock/unlock 오버헤드가 발생. 게임 서버 패킷은 수십~수백 바이트 수준이라 memcpy 비용 자체가 미미하므로 IRP + page lock 오버헤드가 압도적으로 커짐. 루프백에서는 NIC를 거치지 않아 차이가 없었음.

**해결**  
`SO_SNDBUF=0` / `SO_RCVBUF=0` 제거. 커널 기본 송수신 버퍼 사용.

**배운 점**  
Zero-copy는 대용량 전송 환경에서 memcpy가 병목일 때 의미 있는 최적화다. 소형 패킷 고빈도 게임 서버에서는 IRP/page lock 오버헤드가 이득을 압도한다. 최적화 옵션은 항상 실측 후 판단해야 한다.

---

#### Nagle 알고리즘 — LAN 환경 패킷 지터
**날짜**: 2026-04-19

**현상**  
로컬(루프백)에서는 부드럽게 동작하나, 같은 LAN(공유기) 환경에서 다른 클라이언트의 움직임이 심하게 끊김.

**원인**  
Nagle 알고리즘 ON 상태. 작은 패킷이 ACK 대기 중 버퍼링되어 전송 지연 발생. 루프백에서는 레이턴시가 거의 0이라 증상이 없었음.

**해결**  
소켓 옵션 `TCP_NODELAY` 적용.

**배운 점**  
실시간 게임 서버에서는 Nagle을 반드시 끄고 시작해야 한다. 루프백 테스트만으로는 이 문제를 재현할 수 없다.

#### `playerMapChange` 시 플레이어 포인터 댕글링
**날짜**: 2026-04-22

**현상**  
맵 변경 처리 중 `newMap->FindPlayer()`가 nullptr을 반환하거나 해제된 플레이어에 접근하는 문제.

**원인**  
`oldMap->RemovePlayer(sessionID)`가 `unique_ptr`을 erase하면 Player 객체가 즉시 소멸됨. 이후 `newMap->AddPlayer()`를 호출하기 전에 Player 포인터를 사용하거나, RemovePlayer 후 플레이어를 newMap에 추가하지 않아 어느 맵에도 존재하지 않는 상태가 발생.

**해결**  
`GameMap`에 `TakePlayer(SessionID) → unique_ptr<Player>` 메서드 추가. `_players` map에서 erase하기 전 소유권을 로컬 변수로 이전한 뒤 `newMap->AddPlayer()`로 넘김. 소유권 이전이 완료된 후에 DESPAWN 브로드캐스트 수행.

**배운 점**  
맵 간 플레이어 이동은 RemovePlayer가 아닌 TakePlayer로 소유권을 명시적으로 이전해야 한다. `unique_ptr`을 erase로 소멸시키면 포인터를 재사용할 수 없다.

#### 하트비트 timeout unsigned underflow — false positive disconnect
**날짜**: 2026-04-19

**현상**  
정상 플레이 중인 플레이어가 `reason=GameRecvTimeout`으로 Disconnect되며, `data0=18446744073709551600` 같은 비정상 delta 값이 로그에 출력.

**원인**  
`Update()`에서 프레임 시작 시 `now = GetTickCount64()`를 한 번만 읽고 전체 플레이어를 순회. 그 사이 ImmediateThread의 `Dispatch()`가 `lastRecvTimes[]`를 더 최신 시각으로 갱신하면 `lastRecvTick > now`가 발생. `uint64_t recvDelta = now - lastRecvTick` 계산에서 unsigned underflow로 매우 큰 값이 생성돼 timeout 임계치를 초과.

**해결**  
```cpp
if (lastRecvTick <= now)
{
    uint64_t recvDelta = now - lastRecvTick;
    if (recvDelta > TIMEOUT_MS) Disconnect(id);
}
```
역전 가능성을 먼저 방어 후 차이 계산.

**배운 점**  
멀티스레드에서 "현재 시각"과 "최근 갱신 시각"을 다른 스레드가 각각 갱신하면 시간 역전이 발생할 수 있다. timeout 비교는 항상 역전 가능성을 먼저 방어해야 한다.

#### 인벤토리 DB 저장 원자성 — 트랜잭션 처리
**날짜**: 2026-04-22

**현상**  
인벤토리 저장 시 슬롯 단위 INSERT를 단순 반복하면 중간 실패 시 부분 저장 상태가 DB에 잔류.

**해결**  
`START TRANSACTION` → 기존 데이터 `DELETE` → 전 슬롯 `INSERT` → `COMMIT`. 실패 시 `ROLLBACK`. 인벤토리 전체를 한 트랜잭션으로 묶어 원자성 보장.

**배운 점**  
다중 row를 한 단위로 교체하는 경우 반드시 트랜잭션으로 묶을 것.
